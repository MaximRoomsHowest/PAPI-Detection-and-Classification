## Decisions and clarifications captured

### Scope

- **Standard four-light PAPI only.** Alternative PAPI variants
  (APAPI, PLASI) are out of scope for v1.0.
- **Horizontal PAPI detection is a stretch goal**, not a mandatory
  feature. Treat it as a final-presentation bonus.
- The goal triad the client wants demonstrated:
  - Detect the four PAPI lights in a frame.
  - Classify each lamp's state.
  - Recognise red ↔ white transitions as a function of viewing angle
    relative to the lights.

### Software contract — the "two-input join"

- **The angle is supplied by software, not derived from pixels.** The
  surrounding (non-team) software computes the camera-to-PAPI viewing
  angle from the GPS coordinates plus the frame; the team's ML model
  is *not* expected to derive the angle from raw pixels.
- **The team's model must detect the transition in the image and bind
  that transition to the angle value coming from metadata.** This is
  the central per-frame join — `(transition_observed_in_image,
  angle_at_that_frame_from_metadata)`.

### Dataset

- **Coordinates are embedded in the image** — mechanism still to be
  confirmed (overlay text vs EXIF vs sidecar). To be unblocked before
  the loader is written.
- **Two angles / views** of the same installation are present in the
  dataset; the model must be robust across both.
- **Camera information is in the metadata** so the team can
  compensate for camera-specific distortion or interference.
- **Daytime and nighttime** are both provided; both regimes must
  work.
- **Daytime is the hard case.** Sun glare, washed-out red/white
  separation, loss of lamp saturation. Plan augmentation,
  evaluation, and possibly a dedicated daylight handling branch.
- The client acknowledged that **detecting the PAPI lights
  themselves is genuinely hard** — they've had trouble too. The team
  should expect detection recall to be the early bottleneck.

### Architectural implications

- Per-lamp detection + per-lamp state classification + per-lamp
  transition detection. The global five-state can be derived from
  the four per-lamp labels.
- Day vs night likely needs a **conditional handling path** — same
  backbone, but augmentation and evaluation split so the team can
  report daylight metrics honestly instead of letting good night
  performance hide poor day performance.
- The "two angles / views" fact argues for a **viewpoint-aware
  evaluation** — report per-view metrics, not only aggregate.
